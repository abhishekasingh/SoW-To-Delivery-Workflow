SOW → DELIVERY / WORKFLOW SCHEMATIC

Orchestrated by Claude

From approved SoW to weekly stakeholder reporting

One trigger, five downstream stages, three hard confirmation gates. Claude drafts and calculates throughout — Jira writes and outbound email always wait for a human "go."

01–06 — Pipeline stages

01
Task breakdown & hours estimation
SoW → epic/story hierarchy with AI-estimated hours
▸
Input
Approved SoW document
Owner
Claude drafts → PM reviews
  1. Parse the SoW's deliverables and scope statements
  2. Generate an epic → story/task hierarchy, one epic per major deliverable
  3. Estimate hours per task from comparable complexity and any historical velocity supplied
  4. Flag vague scope items ("ongoing support") for PM clarification before sizing
Output: draft ticket list — title, description, hours, epic grouping — shown for review. Nothing is written to Jira from this stage. Every estimate is labeled "AI-estimated, needs PM review."
02
Jira project & sprint creation
Project, tickets, and sprint assignment — each write confirmed
GATE ▸
Input
Approved ticket list + weekly team capacity
Owner
Claude proposes → PM confirms each write
  1. Propose Jira project structure (name, key, template)
  2. Batch approved tickets for creation
  3. Calculate sprint count as total estimated hours ÷ weekly capacity, propose boundaries and assignment
Three separate confirmations: project creation, ticket batch, sprint assignment. No batch write happens without an explicit go-ahead each time.
03
Risk register
Living document, re-evaluated at the start of every sprint
▸
Input
SoW + Stage 1 ticket breakdown
Owner
Claude drafts, PM reviews before external sharing
  1. Draft register: category (scope / technical / resourcing / timeline / vendor) · likelihood · impact · mitigation owner
  2. Re-evaluate each sprint start — add new risks from ticket status or scope change, close resolved ones
Output: version-dated risk register, linked from the weekly status deck.
04
Financial pacing (Tempo)
Logged hours vs. SoW budget, classified into a pacing status
▸
Input
Tempo hours logged + SoW budget figure
Owner
Claude, read-only calculation
  1. Pull logged hours per sprint from Tempo
  2. Compare cumulative burn against budgeted hours/dollars
  3. Classify: on track (±5%) · at risk (5–15% over) · over budget (>15% over)
  4. Surface which epics/tasks are driving variance
Requires the Jira + Tempo connector to be live — the calculation logic is ready, it just isn't wired to real data yet.
05
Weekly status deck
Sprint progress, pacing, and risk rolled into one recurring deck
▸
Input
Sprint data + risk register + pacing status
Cadence
TBD — Friday EOD or Monday AM
  1. Summary — one-line project health, milestone status
  2. Sprint progress — completion %, tickets done/in-progress/blocked
  3. Budget pacing — status + narrative
  4. Risks & blockers — top 3–5, new ones flagged
  5. Next steps — upcoming focus, decisions needed
PM review recommended before the deck goes out, at least for the first few cycles.
06
Stakeholder communication
Separate client and internal drafts — always sent by a human
GATE ▸
Client email
Progress narrative, milestones, qualitative pacing — no internal blocker detail or names
Internal email
Everything above, plus granular blockers, risk detail, action owners, hard numbers
Draft only — a human always sends. This stays true even if the rest of the pipeline becomes more autonomous, since client messaging carries relationship risk Jira tickets don't.
